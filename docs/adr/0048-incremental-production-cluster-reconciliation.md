# Incremental Production Cluster Reconciliation

## Status

Accepted

## Context

The disposable `kind` acceptance lane proves that the complete platform can be created from an empty cluster, but a real production cluster is long-lived and may contain durable state and independently changing workloads. Resetting or clean-deploying that cluster would destroy state, widen the deployment blast radius, and redeploy components unrelated to the actual change.

## Decision

Production delivery must reconcile incrementally against observed cluster state. GitLab CI detects affected components, verifies them, publishes immutable artifacts, and promotes validated desired-state revisions. An in-cluster Argo CD installation continuously compares that desired state with the target cluster and performs the Kubernetes reconciliation. GitLab Runner jobs do not apply production resources directly and do not receive general production-cluster credentials.

Argo CD must preserve unaffected releases and durable data, converge idempotently from the current cluster state, and report drift rather than depending on the next application pipeline run to discover it. A production deployment must never invoke the destructive empty-cluster reset or clean-deploy path.

Application source, context-owned Helm charts, Argo CD resources, and environment desired state remain in this monorepo. Production state lives in protected, reviewable paths separate from application source, but it does not move to a second Git repository. Argo CD watches only those desired-state paths.

GitLab CI computes an explicit change-impact set before publishing or promoting. A context source or chart change affects that context; shared library, contract, CI, platform, and environment changes expand the set through declared dependencies. Components outside the computed set are neither rebuilt nor promoted. The existing service-local `rules:changes` filters remain useful validation/build prefilters, but they are not the complete production-impact model.

The production impact model is a versioned, machine-readable dependency map under `ci/`. It declares each service and platform component, owned paths, shared inputs, contract producers and consumers, chart, desired-state entries, and transitive dependencies. A tested generator compares the relevant Git range, fails closed for unowned changed paths, computes the transitive impact set, and emits a dynamic child pipeline containing only required validation, build, publication, and promotion jobs. Desired-state-only updates validate reconciliation inputs without rebuilding images, while manual pipelines retain explicit component and `all` selection.

Every commit that reaches `main`, whether by direct push or merge-request merge, is eligible for automatic production promotion after its required validation succeeds. CI publishes immutable artifacts only for the computed impact set, then updates only those components in the production desired-state paths. Desired-state-only CI updates must be identifiable and must not recursively rebuild or republish unchanged application artifacts. Argo CD observes the resulting `main` revision and reconciles it without a separate promotion merge request.

Branch protection and mandatory human promotion approval are intentionally deferred for development speed. Direct pushes to `main` therefore have the same production effect as merged changes; this is an accepted temporary governance risk, not a claim that the path is suitable for a larger production team.

Argo CD uses automated synchronization with self-healing for the application resources it owns. A live-cluster mutation that differs from the desired state in `main` is drift and must be reverted automatically rather than preserved until a later pipeline. Emergency runtime changes must therefore be captured in Git immediately if they are intended to persist.

Automated pruning is allowed for replaceable application resources such as Deployments, Services, ConfigMaps, and bounded Jobs. Namespaces, PersistentVolumeClaims, custom-resource definitions, databases, message brokers, and other state-bearing or cluster-scoped resources require explicit deletion confirmation. Removing an Argo CD Application or application release must not cascade-delete production data.

Argo CD manages each microservice as an independently reconcilable Application. Observability, mesh policy, and shared platform infrastructure use separate Applications rather than being coupled to an application release. A versioned ApplicationSet inventory in the monorepo generates those Applications, and each generated Application is scoped to its own desired-state entry and deployment target. Health, synchronization failure, rollback, and ordinary pruning therefore remain component-local.

Production Applications consume immutable Helm packages published by GitLab CI rather than rendering charts directly from moving paths on `main`. Each desired-state entry pins the exact Helm chart version and container image digest for its component. Promotion changes those pins; rollback restores prior pins in Git. A chart-source edit has no production effect until validation, publication, and desired-state promotion succeed.

Production desired state is stored as one small file per independently deployable component under `environments/production/`. Service entries live under `environments/production/services/`; in-cluster managed platform releases such as ESO mappings and observability live under `environments/production/platform-releases/`; external datastore capability contracts remain under `environments/production/platform/`; and the Argo CD root, projects, and ApplicationSets live under `environments/production/argocd/`. Each managed release entry declares its chart identity and version, image repository and digest when applicable, destination namespace, and production value inputs. CI updates only entries in the computed impact set and refuses to release production platform entries that still contain placeholder coordinates or configuration.

The local simulator has a separate `environments/local-production/` inventory with the same component-file schema and ApplicationSet template. Service promotions carry the same tested chart versions and image digests into both eligible inventories, while environment entries select different external-resource endpoints, provider configuration, replicas, requests, limits, and storage durability. Argo CD in `uno-arena-production` watches only `local-production`; it cannot apply the real `production` inventory to the simulator cluster.

Simulator-only Postgres, Kafka, Redis, KurrentDB, ClickHouse, Keycloak, and MinIO dependencies are independently tracked Argo CD platform Applications rather than imperative bootstrap steps. Kubernetes runs their ordinary Pods, Services, and PersistentVolumeClaims in the simulator cluster; Argo CD only reconciles the declared resources through the Kubernetes API. Local stateful dependencies use persistent volumes so ordinary Pod replacement and incremental synchronization preserve data, while remaining explicitly reduced-resource and non-HA. The real production inventory omits these local deployments and references externally managed services.

Production application services and their in-cluster dependency endpoints remain in namespace `uno-arena`, matching the existing production DNS contract. The observability platform remains in `observability`, and Argo CD runs in `argocd`. The checkpoint-only `staging` namespace is not a production deployment target. Moving to per-service namespaces is a separate security-hardening migration because it changes service discovery and mesh authorization contracts.

Argo CD is introduced through a one-time, operator-run bootstrap owned by this repository. The bootstrap pins the Argo CD version, refuses any Kubernetes context other than its explicit target, creates namespace `argocd`, installs only a restricted bootstrap AppProject plus root Application, and configures read access to this GitLab repository and its Helm registry. The root Application then owns the environment projects, ApplicationSets, itself, and—for the simulator—the repository-rendered foundation Application. GitLab Runner does not receive cluster credentials to perform the bootstrap. Afterward, Argo CD incrementally manages and self-heals its repository-owned configuration; pruning remains disabled at the root and foundation boundaries.

The GitLab repository, Helm package registry, and container registry are treated as private. Bootstrap accepts project-scoped, read-only machine credentials through environment or interactive input and creates the required Kubernetes Secrets without writing their values to Git. Argo CD receives repository and Helm-package read authority; application workload service accounts receive a separate image-pull credential. GitLab CI uses job-scoped credentials for publication. Registry credentials must be attached to the service accounts used by the Pods rather than only to the namespace's `default` service account.

Application runtime secrets are reconciled through External Secrets Operator. Git contains `ExternalSecret` definitions, provider references, and stable target Secret/key contracts, but never secret values. Production uses an external secret-manager provider. The local production simulator uses the same workload-facing `ExternalSecret` and target Secret names with a simple local provider substitute; only provider configuration and resource sizing differ. Argo CD owns the definitions, External Secrets Operator owns target Secret materialization, and application Helm charts consume the resulting stable Kubernetes Secret names.

The simulator's local provider uses a dedicated `secret-seed` namespace populated only by the operator bootstrap from environment variables or an ignored local input file. A dedicated External Secrets Operator identity may read source Secrets only in that namespace. Workload `ExternalSecret` resources select only declared keys into their own target namespaces; application identities cannot read `secret-seed`. Target Secrets are disposable projections and may be restored by the operator without rerunning GitLab CI.

The delivery workflow must be proven on this machine with a long-lived local production simulator. It uses the production GitLab, immutable-artifact, desired-state, Argo CD, namespace, reconciliation, drift, and rollback semantics, while local overlays may reduce replicas, requests, limits, and storage durability. The current disposable cluster may be removed once to establish that simulator; ordinary simulator deployments are incremental and must not reset it.

The simulator uses kind cluster `uno-arena-production` and kubectl context `kind-uno-arena-production`. It must never be targeted by the destructive `uno-arena` acceptance scripts. The existing disposable `uno-arena` cluster may be deleted during initial setup; recreating or resetting that acceptance cluster remains a separate explicit operation.

The simulator topology is one kind control-plane node plus two worker nodes. Replicated application workloads must use scheduling constraints that exercise worker-node spreading and rolling replacement. Local stateful substitutes remain single-replica and explicitly non-HA, but use PersistentVolumeClaims so Pod replacement and ordinary Argo reconciliation preserve their data. The simulator makes no multi-zone or production-storage claim.

The simulator exposes the existing BFF-only public edge through Argo-managed Kubernetes Gateway API resources. Only the Gateway service is published; application services and datastores remain ClusterIP-only. A stable host entrypoint replaces manual `kubectl port-forward`, and integration tests use that entrypoint without an operator-maintained side process. This preserves the existing single public API boundary while making external routing declarative and independently reconcilable.

Cert-manager is an independently reconciled platform Application. Gateway API terminates public TLS using a cert-manager `Certificate`; the simulator uses a local issuer and `uno-arena.local`, while production supplies an environment-owned issuer through the same contract. The local HTTPS entrypoint uses host port `8443`, and host port `8080` redirects to HTTPS. East-west Istio Ambient mTLS remains a separate transport boundary. Forwarded client identity is accepted only from the controlled Gateway API proxy path so edge rate limiting does not trust arbitrary client-supplied forwarding headers.

Repository changes, offline validation, Argo bootstrap, cluster readiness, drift correction, and incremental reconciliation are verified before GitLab execution. The final GitLab pipeline run is a separate acceptance step initiated by the user after starting the local runner; lack of a live runner does not justify weakening or bypassing any earlier gate.

GitLab publication is not complete when desired state is written. Using a read-only Argo CD API credential rather than kubeconfig, the pipeline waits for every Application in the computed impact set to become Synced and Healthy and for its required in-cluster post-deploy evidence to pass. Per-component checks cover readiness and critical local behavior; cross-context impacts select the relevant integration probes; Gateway or public-contract impacts run client parity through the HTTPS edge; observability and platform impacts run their telemetry and health checks. Degraded, out-of-sync, timed-out, or failed-probe states fail the pipeline.

When an affected stateless/service Application fails synchronization, readiness, or required post-deploy evidence within its bounded window, automation creates a Git rollback commit that restores that component's previous known-good chart version and image digest. The failed pipeline remains open until Argo CD reports the restored revision healthy or the rollback itself fails. PersistentVolumeClaims, databases, brokers, custom-resource definitions, stateful platform Applications, and irreversible migrations are never rolled back automatically. Schema changes must follow expand/contract compatibility when application rollback is required; otherwise deployment stops for operator intervention.

The initial progressive-delivery baseline uses safe Kubernetes rolling updates rather than canary traffic splitting. Replicated APIs declare zero unavailable capacity, bounded surge, startup probes aligned to initialization, readiness-first removal, pre-stop draining, termination grace longer than application shutdown, PodDisruptionBudgets, and worker-node topology spreading. Argo health plus post-deploy RED evidence gates completion. Argo Rollouts and metrics-based canaries remain deferred until the new service RED metrics and alerts have established trustworthy thresholds; a canary without reliable analysis signals is not considered safer.

Autoscaling remains deliberately simple. Metrics-server supplies resource metrics, and replicated HTTP APIs use bounded `autoscaling/v2` CPU HorizontalPodAutoscalers with environment-specific minimums and maximums. Workers, controllers, Room runtime Pods, and stateful dependencies retain explicit fixed scaling in this phase. Kafka-lag, active-SSE, and other workload-specific scaling remain documented production hardening rather than introducing KEDA or a custom metrics adapter into the simulator.

Synchronous failure containment remains application-owned. Shared bounded deadline and per-upstream circuit-breaker primitives expose observable closed, open, and half-open behavior while preserving domain-specific uncertain-write handling. Mutations and ambiguous writes are never transparently retried; only explicitly idempotent reads may use small bounded retry policies. Istio Ambient remains responsible for transport mTLS and exact service-account-to-service authorization, replacing namespace-wide application trust, without introducing L7 waypoints or mesh retry policy in this phase.

The observability platform preserves existing structured logs, distributed traces, business counters, and trace-to-log navigation while adding service RED dashboards, alert rules, Alertmanager routing, and self-monitoring for Alloy, Loki, Tempo, Prometheus, Redis, and Argo CD. The simulator routes notifications to a small in-cluster webhook sink and proves delivery. Production obtains its environment-owned notification endpoint through External Secrets Operator. Notification-path failure is observable but never becomes an application readiness dependency.

Schema changes are an ordered, immutable migration history. The migration runner verifies a contiguous release and exact live prefix, rejects checksum drift and non-prefix history, and records each version/checksum itself only after the SQL succeeds; post-initial migration files may not mutate the runner-owned ledger. PostgreSQL applies pending versions in one advisory-locked transaction. ClickHouse records explicit running/succeeded attempts around idempotent statements and fails closed on incomplete history. Application processes do not own schema migration, and migrations must follow expand/contract compatibility.

The current context-bootstrap chart runs these jobs as Argo CD PreSync hooks in its separate local-production platform Application. That proves incremental data preservation and blocks that platform sync, but it does **not** yet establish an Argo ordering edge between the separate context-bootstrap Application and a service Application promoted in the same revision. No production context-bootstrap release is enabled. Until a production-safe per-context migration Application or equivalent ApplicationSet dependency is implemented and live-proven, production service promotion that includes schema change remains disabled/fail-closed operator work; CI must not infer safe service-before-migration ordering from sync waves across Applications.

The `kind-uno-arena` empty-cluster lane remains the canonical reproducibility and full-system acceptance proof defined by ADR-0045; it is not the production deployment algorithm.

## Consequences

Production delivery requires explicit cluster identity, serialized ownership of each deployment target, immutable release inputs, a versioned dependency/impact map, preflight inspection, readiness and post-deploy evidence, and a rollback path that does not reset the cluster. Stateful infrastructure lifecycle and application release lifecycle must remain separate. Argo CD holds narrowly scoped reconciliation authority; GitLab holds registry and same-repository desired-state publication authority, keeping build credentials separate from cluster credentials. Same-repository promotion must also prevent CI-authored desired-state updates from recursively rebuilding unchanged application artifacts.
