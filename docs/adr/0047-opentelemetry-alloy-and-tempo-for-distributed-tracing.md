# OpenTelemetry, Alloy, and Tempo for Distributed Tracing

## Status

Accepted

## Context

Correlation-aware logs are useful but do not show the timing and causal structure of a request across synchronous service calls, Kafka delivery, database work, and background workers. Deployment acceptance requires distributed tracing as part of the complete observability platform. Introducing both Grafana Alloy and a separate OpenTelemetry Collector would duplicate the collection layer.

## Decision

Instrument repository-owned Go services and workers with OpenTelemetry SDKs and propagate W3C Trace Context across HTTP and Kafka headers. Applications export OTLP to a dedicated Alloy gateway Deployment in the `observability` namespace. Its private ClusterIP Service exposes OTLP/gRPC `4317` and OTLP/HTTP `4318`, with applications defaulting to gRPC. Istio Ambient strict mTLS authenticates exporters, and default-deny authorization allows those ports only from the dedicated application service accounts in `uno-arena`; matching NetworkPolicy applies where enforced. No bearer token or shared telemetry secret is introduced. Alloy receives, batches, and processes spans, then forwards them to Grafana Tempo; no separate OpenTelemetry Collector is deployed. The Alloy DaemonSet remains responsible for node-local log collection.

The Alloy roles use different workload identities. `alloy-logs` owns the DaemonSet and receives only read-only pod metadata access plus required host-log mounts. `alloy-otlp` owns the gateway, disables service-account token automount, and has no Kubernetes API permissions. Loki and Tempo accept ingestion only from the corresponding Alloy/backend identities rather than directly from application pods.

Local `kind` deploys monolithic Tempo and stores traces in a dedicated `tempo-traces` bucket on the existing 5 GiB S3-compatible MinIO PVC shared with Loki's separate bucket. Tempo enforces 24-hour trace retention and owns a separate 1 GiB PVC for its WAL and local working data. Its single-node overlay cuts live-store blocks every ten minutes to avoid object-store churn during the acceptance burst, while polling the object-store block list every minute so completed blocks become queryable within bounded persistence acceptance. The local Alloy-to-Tempo exporter absorbs transient node contention with a bounded 2,048-request in-memory queue, two consumers, a 30-second request timeout, and retries capped at ten minutes; it does not make application exporters blocking. MinIO and Tempo pod replacements preserve accepted traces; deleting the kind cluster intentionally deletes both PVCs. Tempo owns trace deletion, with no competing local MinIO lifecycle rule. Production defaults to an external S3 bucket and may scale Tempo's deployment topology without changing application OTLP instrumentation or the Alloy gateway contract. Grafana provisions Tempo as a third data source and links traces to Loki through `trace_id` and `span_id` fields in structured logs.

The portable trace/log/metrics platform is delivered by the repository-owned `infrastructure/observability/helm/uno-arena-observability` chart. Kind first applies and readies its local-only MinIO/PVC/bucket foundation, then installs the chart with `values.kind.yaml`, and only then deploys applications. Production values reference external S3 configuration and Secrets without provisioning cloud resources.

Instrumentation covers inbound and outbound HTTP, Kafka producer and consumer operations, database operations, and important application spans around command handling, durable commit, publication, and worker processing. Trace attributes follow OpenTelemetry semantic conventions and never contain credentials, private gameplay state, raw payloads, or unbounded data promoted to metrics or Loki labels.

Adapter construction uses established instrumentation libraries: `otelhttp` for inbound/outbound HTTP, franz-go `kotel` for direct Kafka clients, `redisotel` tracing for Redis clients, and `otelpgx` for `pgxpool`; Analytics wraps its existing ClickHouse HTTP transport with `otelhttp`. Game Integrity defines manual spans around its context-owned KurrentDB append, read, and expected-revision operations rather than relying on opaque internal gRPC spans. Each bounded context manually defines its command, durable-commit, publication, and worker spans. Instrumentation uses bounded route/operation names and excludes SQL text, Redis keys, raw payloads, credentials, and domain identifiers from metric labels.

Every process uses the same resource identity: canonical stable `service.name`, `service.namespace=uno-arena`, immutable build/commit `service.version`, `deployment.environment.name`, Kubernetes pod UID as `service.instance.id` with hostname fallback outside Kubernetes, and bounded `unoarena.component` for the API or worker role. Roles do not mint separate service names. Domain and correlation identifiers are never resource attributes or metric labels; where policy permits them, they remain span/log fields. Prometheus scrape metadata supplies Kubernetes target identity without adding every resource attribute to each application series.

Cross-service SDK mechanics live in a separate infrastructure-only `platform/telemetry` Go module. That module owns resource construction, propagation, trace and metric provider setup, OTLP and Prometheus exporter wiring, bounded queues, shutdown flushing, and construction of the standard `log/slog` JSON handler. The handler applies the shared resource identity, protects reserved base fields, and automatically adds `trace_id` and `span_id` from the active context. Logs remain one-line JSON on stdout for Alloy collection; there is no OTLP log exporter. The module contains no domain types or business metric catalog. Each bounded-context service defines its own bounded event names, optional messages, domain fields, sanitization, log placement, instruments, span names, attributes, and recording seams locally, including the post-commit business-counter hooks. The existing `shared` module remains limited to its standard-library cross-context helpers rather than becoming an observability dependency hub.

Context-owned rejection-audit adapters may call the runtime logger handler's `Handle` method directly to retain fail-closed error propagation while reusing the exact JSON schema and active trace correlation. This does not move audit idempotency, conflict rules, or domain record meaning into the telemetry module. Required Kubernetes workloads use stdout only; file-backed audit sinks remain an explicit offline/test facility.

The module presents one deep interface: `Start(ctx, config)` returns a runtime exposing the logger, explicit tracer and meter providers, and text-map propagator; `runtime.Shutdown(ctx)` performs bounded teardown. A process starts it exactly once before constructing instrumented HTTP, Kafka, Redis, or Postgres adapters, and callers pass providers and propagators explicitly. The sole global provider exception is the same runtime MeterProvider, registered once for the process lifetime through OpenTelemetry's standard global meter seam so SDK self-observability can emit through the Prometheus reader. No global TracerProvider or propagator is installed. The standard global OpenTelemetry error handler writes sanitized structured exporter failures. Production does not swap providers during shutdown; singleton behavior is tested in isolation. Start validates required configuration and synchronously binds the metrics listener before returning. Disabled mode returns no-op tracing/metrics with the same JSON logger. Exporter factories, registries, queues, and metrics-server details remain internal seams. During termination the workload stops accepting and processing business work before telemetry shutdown performs one bounded flush.

For CDC-mediated publication, every Kafka-bound transactional outbox and Room's realtime outbox stores nullable W3C `traceparent` and `tracestate` columns atomically with the domain transition. Debezium's Outbox Event Router maps them to standard Kafka headers, while Debezium Server places them in Redis entry metadata for Gateway extraction. Direct Kafka producers inject the same headers and consumers extract them before creating consumer spans. W3C baggage is not persisted or propagated through outboxes because it is unbounded and may carry sensitive data. Since no environment has been deployed, the initial schemas are corrected directly without upgrade migrations.

The live acceptance trace starts at the Gateway inbound span for the accepted casual-game command that produces `GameCompleted`. It must include Gateway-to-Room HTTP client/server spans, Room command handling and authoritative commit, Game Integrity append, the Room transactional-outbox write, Debezium-mediated Kafka publication, and Ranking consumption plus durable commit. Debezium is responsible for transporting the stored W3C context into Kafka headers and is not required to emit its own span. Tempo must return that complete causal trace, while Loki must return its structured logs by both `trace_id` and the command `correlationId`; neither identifier becomes a Loki label. On the single-node kind profile, business bots pace known-result gameplay mutations by two seconds while retaining a zero action interval for ordinary CLI use. Between independent flows, observability readiness and a subsequent 15-second exporter settle combine with that pacing and the kind Gateway's bounded twelve-second backend timeout; no one delay is treated as proof that telemetry was delivered.

Local `kind` uses the custom `x_parentbased_mutations` head sampler at the trusted Gateway. It always samples accepted `POST`, `PUT`, `PATCH`, and `DELETE` roots so command acceptance has deterministic complete traces, while `GET` and other autonomous roots use the production-like `0.001` (0.1 percent) ratio. Other long-running services use configurable `parentbased_traceidratio=0.001` for autonomous root work while consistently following the parent decision across HTTP and Kafka. Short-lived, repository-owned rebuild acceptance Pods may use `parentbased_always_on` for deterministic evidence. The Gateway establishes the root sampling decision and does not allow an untrusted external client's sampled flag to force collection. This avoids tracing every safe-state polling read during the mandatory best-of-three flow on the single-node acceptance cluster without weakening mutation-trace evidence. The local Alloy gateway batches spans but does not tail-sample them.

Telemetry export is never a synchronous business dependency. Services use bounded batch processors, queues, retry windows, export timeouts, and shutdown flushes. Queue saturation or an Alloy/Tempo outage drops spans and emits sanitized self-observability signals rather than blocking commands, HTTP responses, Kafka processing, or authoritative commits. A batch held over an Alloy outage may still be dropped through its stale gRPC connection after the replacement becomes Ready; recovery acceptance therefore issues bounded fresh commands until one new complete trace proves exporter reconnection. Application readiness does not ping Alloy or Tempo, although the deployment acceptance gate requires the complete observability stack before running proofs.

The single-node `kind` overlay retains Tempo's S3-backed immutable blocks and 24-hour retention but defers compaction measurement and scheduling for one hour, outside the expected acceptance window. Compaction is maintenance rather than an ingestion or durability prerequisite; postponing it prevents local object-store rewrites from competing with the instrumented game flow on the same node. The base and production chart keeps the normal one-minute measurement and 30-second minimum scheduling cadence.

Required mode enables the pinned SDK's official opt-in `OTEL_GO_X_OBSERVABILITY=true`. Its nonblocking BatchSpanProcessor uses queue `2048`, export batch `512`, schedule delay `5s`, export timeout `5s`, no `WithBlocking`, and OTLP retries bounded by the same export deadline. Queue capacity, queue size, processed spans, and `queue_full` drops are exposed through the Prometheus MeterProvider. These SDK self-metrics are experimental internal operational signals, not business contracts; exact dependency pins and offline exposition tests surface changes. A local exporter wrapper records a bounded export-failure counter and sanitized structured error because the SDK queue metrics do not classify backend export failures.

Telemetry startup uses explicit `disabled` and `required` modes. Direct/offline execution may opt into `disabled`; every Kubernetes Helm workload, including APIs and workers, must render `required`, and acceptance rejects disabled or unspecified telemetry. In `required` mode, malformed resource identity, invalid exporter configuration, or failure to bind the metrics listener is a process-startup error rather than a silent degraded mode. Once initialization succeeds, Alloy or Tempo reachability never changes application readiness or blocks business work; bounded export failure behavior applies instead.

The configuration contract uses `TELEMETRY_MODE`, the existing `SERVICE_NAME` and `DEPLOYMENT_ENV`, immutable `SERVICE_VERSION`, bounded explicit `UNOARENA_COMPONENT`, Downward-API `POD_UID`, standard `OTEL_EXPORTER_OTLP_ENDPOINT`, `OTEL_EXPORTER_OTLP_PROTOCOL`, `OTEL_TRACES_SAMPLER`, and `OTEL_TRACES_SAMPLER_ARG`, and `METRICS_ADDR=:9090`. Helm injects the complete required contract into APIs, service-less workers, and dynamically created Room runtime pods. Component identity is explicit and allowlisted rather than derived from pod names or arbitrary worker-role text.

All repository modules, including `platform/telemetry`, use language level `go 1.26.0` and toolchain `go1.26.5`. Exact direct observability pins are OpenTelemetry core/trace/metric/SDK/SDK-metric and OTLP/gRPC exporter `v1.44.0`, Prometheus exporter `v0.66.0`, `otelhttp v0.69.0`, franz-go `kotel v1.7.0`, `redisotel/v9 v9.21.0`, `otelpgx v0.11.1`, and Prometheus Go client `v1.23.2`. Existing Redis clients align on `go-redis/v9 v9.21.0`; existing `pgx/v5 v5.10.0` and franz-go `v1.21.5` remain. A dependency's older `go` declaration is its minimum supported language level, not the repository build target. These source dependencies are compiled into repository images for the selected node architecture and do not constrain the independently pinned ARM64/AMD64 upstream OCI images.

## Consequences

The empty-cluster lane must install and ready MinIO, Tempo, and the Alloy OTLP gateway; validate the Gateway-rooted local sampling policy, 24-hour trace retention, 5 GiB shared MinIO claim, and 1 GiB Tempo WAL/work claim; reject every Kubernetes workload that does not require telemetry; produce a multi-service acceptance trace that crosses HTTP plus Debezium-mediated Kafka or Redis delivery; query it through Tempo; prove it survives Tempo and MinIO pod replacement; and verify Grafana trace-to-log navigation without manual configuration. Offline contract checks require the two nullable outbox columns and exact Debezium placements while rejecting baggage persistence. A live failure-path probe removes the Alloy gateway, proves gameplay continues without readiness failure or unbounded blocking, restores Alloy, and issues bounded fresh post-recovery commands until one complete new trace proves reconnection; trace loss during the bounded outage and from a held stale connection is acceptable and observable.

Local images pin exact releases plus verified Linux AMD64/ARM64 OCI index digests: Alloy `v1.17.1` at `sha256:4f6ddc56ffdcf8a6316748fc5162972e20cb301523cac1bb4a31957df733ae9b`, Tempo `3.0.2` at `sha256:cda87c212d8c584dc0b89e337e7ed648a5100feb657e5d528480ee4fa03dbbe3`, and Grafana `13.1.0` at `sha256:121a7a9ece6dc10b969f1f96eed64b4f07dfac0d0b8abc070f7cb83bbde86f63`. Workloads reference the immutable index directly so the node runtime selects its Linux AMD64 or ARM64 child.
