# Kind as Empty-Cluster Deployment Acceptance

## Status

Accepted

## Context

The deployment evaluation starts from an empty Kubernetes environment and requires the complete platform, including observability and business metrics, to be installed and proven. It does not require a particular cloud, managed Kubernetes product, or CPU architecture. The repository already has a disposable `kind` lane, while Docker Compose and offline structure checks cannot prove Kubernetes deployment completeness.

## Decision

Use the pinned local `kind` cluster `uno-arena` as the canonical deployment-acceptance environment. The explicit acceptance command owns that disposable cluster's lifecycle and starts from no cluster: it creates the cluster, builds and loads repository images, installs all platform infrastructure, services, and observability components, waits on real readiness conditions, and runs functional plus business-metric probes. Starting from no cluster is the stronger reproducibility proof; the underlying apply and deploy stages must also remain usable against an already-created empty cluster.

Deployment order is foundation and Istio, local-only MinIO plus an idempotent `loki-logs`/`tempo-traces` bucket Job, the repository-owned `uno-arena-observability` Helm release, and then the application-context Helm releases. Every stage is readied before its dependents. Bringing the collectors and backends up before applications preserves startup logs and makes telemetry infrastructure available before workload initialization and functional probes. A cold-cache Istio installation has a ten-minute local timeout so exact pinned image pulls remain part of the empty-cluster proof instead of depending on a warmed node; the value remains explicitly overrideable. The kind overlay lets the Alloy Logs DaemonSet burst to 500m CPU while it discovers and initializes readable node log files; production retains the base chart resource policy.

The deliberate Alloy OTLP outage proof separates availability from recovery. It first proves an authoritative gameplay mutation while the collector has no pods. After restoring Alloy, it waits up to five minutes for every observability workload to regain readiness before generating recovery traffic, allowing the bounded simultaneous exporter reconnect burst to drain on the shared node. It then requires a fresh complete trace within its independent three-minute evidence window. This recovery gate does not couple application readiness to telemetry and does not weaken the during-outage gameplay proof.

Docker Compose is not an acceptance environment. Offline rendering and structure checks remain useful preflight gates, but only the live empty-cluster lane proves deployment. A successful local `kind` run proves Kubernetes completeness and reproducibility on the development machine; it does not claim production high availability, capacity, durability, or cloud portability beyond standard Kubernetes and Helm interfaces.

Generic upstream workloads pin exact releases plus immutable OCI multi-architecture index digests containing Linux AMD64 and ARM64 children; the node runtime resolves the matching child. The lane does not embed or pre-stage an ARM64 child in generic manifests. Repository-built application images are built and loaded for the detected node architecture. A required image without a suitable index uses an explicit architecture-to-digest map and fails closed for unknown architectures; KurrentDB is the current documented exception. Application Helm resources inject the canonical OpenTelemetry `service.name`, `service.namespace`, `service.version`, `deployment.environment.name`, pod-UID `service.instance.id`, and bounded `unoarena.component` values through explicit configuration and the Downward API; acceptance rejects identifier-bearing resource attributes or metric labels.

The single-node kind overlay is an explicit local execution profile rather than a production sizing claim. Repository-owned API and worker containers retain 50m/64Mi requests for scheduling but may burst to 500m CPU and 256Mi memory; their HTTP health probes use a three-second timeout, require six consecutive failures before restart, and run every 20 seconds. Controller-created Room runtime Pods receive the same kind-only CPU ceiling and probe tolerance through the runtime-controller configuration. The four local Postgres processes and Redis use 15-second readiness and 30-second liveness checks, while Debezium Connect uses 20- and 30-second checks, avoiding continuous kubelet exec-process churn on the shared node. The local business bots poll every five seconds by default, and the lane runs their business-metric and canonical-trace proof immediately after the observability baseline and connector readiness checks, before adapter and projection-rebuilder probes accumulate single-node load. Between its independent casual and tournament flows, the lane requires every observability component to recover readiness after draining telemetry and then retains the 15-second settle window. Bot setup boundedly retries its authenticated identity GET under transient Gateway pressure, while every mutation remains single-attempt and unknown mutation outcomes are never replayed. The 189-test Tournament store integration lane caps Go test parallelism at two and has a five-minute package budget so intentional Redis-fallback retries plus real Postgres checks remain bounded without starving the local node. These acceptance controls remain local and change no production service timeout or client default. Base, staging, and production values retain their independently reviewed resource and probe defaults, including the ten-second application health cadence.

## Consequences

The acceptance lane must fail closed when any required image, infrastructure component, service, observability component, readiness gate, functional probe, or business-metric proof is missing. It installs Istio Ambient, enrolls both `uno-arena` and `observability`, enforces strict mTLS and explicit L4 authorization, and proves that the Prometheus service-account principal can scrape application port `9090` while an unauthorized workload principal cannot. Kind's plaintext exception applies only to documented application-layer database TLS and disposable credentials, never to east-west mesh transport. Its observability installation runs in a dedicated `observability` namespace with component-specific least-privilege service accounts and includes the ADR-0046 Alloy-to-Loki log pipeline, ADR-0047 OpenTelemetry/Alloy/Tempo tracing pipeline, and local MinIO S3-compatible storage, plus standalone Prometheus scraping service-owned OpenTelemetry Prometheus-exporter `/metrics` endpoints on named port `metrics` (`9090`), kube-state-metrics, authenticated kubelet/cAdvisor resource scraping, and stateless Grafana provisioned with Loki, Prometheus, and Tempo data sources plus the required dashboards. Prometheus uses EndpointSlice discovery through existing Services for stable APIs and direct pod discovery for service-less workers and dedicated Room runtimes. Mutually exclusive scrape modes prevent duplicate targets; no metrics-only worker Service or Prometheus Operator custom resource is introduced, and unready pod-discovered workloads remain observable. All local scrape and evaluation intervals are 15 seconds with a 5-second scrape timeout; production may override them. Application metrics use the OpenTelemetry Meter API and a process-local custom Prometheus registry with an explicit underscore-and-type-suffix translation; Alloy is not in the metrics path. The process-local business counters are operational evidence rather than authoritative accounting; context-owned durable state and events remain authoritative, and dashboards aggregate replicas and resets with `sum(increase(<counter>[$__range]))`. The counters use unit `1` and have no instrument labels; target discovery supplies bounded service, namespace, component, pod, and instance labels. Local Prometheus uses a 5 GiB PVC, 24-hour retention, and a storage ceiling with filesystem headroom. Local tracing uses the trusted Gateway's parent-based mutation sampler: command roots are always sampled for complete public acceptance evidence, read roots use the production-like 0.1 percent ratio, and downstream services inherit the Gateway decision; short-lived repository-owned rebuild probes may sample always-on. Acceptance must wait up to 45 seconds for a pre-flow baseline scrape, run the corresponding CLI flows, wait up to 45 seconds for a post-flow scrape, and prove a positive increase for each exact `unoarena_players_created_total`, `unoarena_games_completed_total`, and `unoarena_tournaments_completed_total` series within the bounded post-baseline interval. It must also prove a structured application log is queryable by bounded labels and parsed correlation field without sensitive data, prove a complete multi-service trace crosses HTTP plus a Debezium-mediated Kafka or Redis hop and links to its logs, prove declared telemetry history survives its pod-replacement scenarios, verify provisioned data sources and dashboards through Grafana's HTTP API, and fail if observability identities can read application Secrets. Offline exposition contract tests pin each required series name, counter type, unit `1`, exact help text, and absence of instrument labels across OpenTelemetry SDK/exporter upgrades. Offline checks also require nullable `traceparent`/`tracestate` outbox fields and exact connector placements while rejecting baggage persistence. A live failure probe removes Alloy, proves gameplay and application readiness remain healthy under bounded exporter failure, restores Alloy, and proves new traces resume. Grafana remains ClusterIP-only and requires no manual UI configuration. Cluster deletion remains explicitly confirmed and scoped to `uno-arena`; teardown after a successful run is separate so evaluators can inspect the running system.

Tempo's kind-only one-hour compaction measurement and minimum-cycle intervals pair with a three-hour scheduler work-prune age. This preserves Tempo's required relationship that tenant measurement occurs at least twice before scheduler work is pruned while keeping compaction outside the short acceptance window. Base and production values use one-minute measurement, a 30-second minimum cycle, and a one-hour work-prune age.

The required cross-service trace is specifically the accepted casual-game command that emits `GameCompleted`: Gateway inbound and Gateway-to-Room HTTP, Room command/commit, Game Integrity append, Room outbox, Debezium-mediated Kafka context transport, and Ranking consume/commit must share one causal trace. Debezium need not emit a span. Acceptance queries the trace through Tempo and its structured logs through Loki by both `trace_id` and `correlationId` without identifier labels.

Destructive persistence acceptance captures each current Ready backend pod by exact name before replacement. Its bounded retries cover only idempotent desired-state Kubernetes mutations and retain that exact identity, because a timed-out API request may already have committed; label-based delete retries could otherwise remove the newly created replacement.
