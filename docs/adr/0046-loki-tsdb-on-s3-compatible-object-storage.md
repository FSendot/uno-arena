# Loki TSDB on S3-Compatible Object Storage

## Status

Accepted

## Context

Centralized logs must be deployable and queryable in the empty-cluster acceptance lane without creating a filesystem-only local architecture that diverges from production. Grafana Alloy is the collection and forwarding layer, not the durable query backend. Loki requires durable object storage for the selected production-shaped path.

## Decision

Use Grafana Alloy to collect Kubernetes workload logs and forward them to a monolithic Grafana Loki deployment. Loki uses the TSDB index and an S3-compatible object-store contract for both local and production environments.

Loki retains its WAL disk-usage backpressure. Production uses the conservative `0.90` threshold; the disposable kind overlay uses `0.99` because its container node shares the local engine's image filesystem and can begin above 90 percent while retaining several GiB for the bounded acceptance lane. Disk protection is not disabled.

The log DaemonSet runs as `alloy-logs`, with only read-only pod metadata access and the required read-only host-log mount. Kubernetes container runtimes expose `/var/log/pods` as `0750 root:root` node state, so the non-root UID/GID 473 collector receives supplemental GID 0 only to traverse and read that mount; it retains `runAsNonRoot`, `RuntimeDefault` seccomp, a read-only root filesystem, disabled privilege escalation, and all Linux capabilities dropped. No application or backend workload receives the supplemental group. The separate trace gateway runs non-root as `alloy-otlp`, disables token automount, and has no Kubernetes API permissions. Loki ingestion is private and mesh-authorized for the log-collection/backend identities rather than application pods; applications never send logs directly to Loki or through OTLP.

The local `kind` environment deploys MinIO as the S3-compatible object store, provisions dedicated `loki-logs` and `tempo-traces` buckets, and stores both on one 5 GiB Kubernetes PersistentVolumeClaim. Monolithic Loki enables its WAL, explicitly configures the TSDB shipper's active-index and query-cache directories, and flushes on graceful shutdown. Acceptance creates a fresh player exactly once through the public CLI with a dedicated low-volume correlation marker. An unknown registration is never replayed; after a successful registration, only an authenticated `whoami` read may be repeated with the same marker until Loki proves the recovered application-log path. Acceptance then uses the supported ingester shutdown with `flush=true` and full-process termination. That graceful process stop cuts the active 15-minute TSDB head and synchronously gives the index shipper its final upload opportunity; the probe waits for kubelet to restart the same Pod, then deletes that exact Pod identity so the distinct replacement has neither the local WAL nor active index. That replacement must recover the correlated log from S3 before MinIO is replaced. After MinIO itself is Ready again, acceptance restarts Loki once more and performs the final query through this fresh process, proving that neither a warm Loki cache nor its earlier local index view masks failure to recover through the replaced S3 endpoint. Tempo separately preserves the canonical cross-context gameplay trace. Local objects survive MinIO pod replacement while the cluster exists. Loki's Compactor enforces a 24-hour local log-retention period; Tempo independently enforces 24-hour trace retention, and no competing local MinIO lifecycle rule deletes either backend's objects. Deleting the disposable `uno-arena` kind cluster intentionally deletes the volume and its telemetry; the acceptance lane does not depend on host-mounted or external state. Production uses external S3 buckets by default. Loki's storage schema and S3 API remain the same across environments; environment configuration supplies the endpoint mode, bucket, region, TLS settings, credentials, encryption, and retention/lifecycle policy. Local MinIO credentials, custom endpoint behavior, capacity, and retention are never production defaults.

MinIO, its PVC, and its idempotent bucket Job are local-only kind foundation manifests. After that Job succeeds, the repository-owned `infrastructure/observability/helm/uno-arena-observability` chart installs the portable Alloy, Loki, Tempo, Prometheus, kube-state-metrics, Grafana, policy, data-source, and dashboard resources. `values.kind.yaml` targets MinIO; `values.production.yaml` carries only external-S3 settings and Secret references and never provisions cloud storage.

Repository-owned Go services and workers emit one JSON object per line through `log/slog`. Every line has UTC RFC3339Nano `timestamp`, lowercase `level`, `service`, `component`, `environment`, `version`, `instance_id`, and a bounded snake-case `event`; `message` is optional human text. Valid active spans add `trace_id` and `span_id`. Domain and correlation fields retain their canonical contract spelling, including `correlationId`, `commandId`, `eventId`, `roomId`, `gameId`, and `tournamentId`. Sanitized failures use only `error`, eliminating an `err` alias. Caller attributes cannot override reserved base fields, and logs omit source paths, stack dumps, credentials, tokens, private hands, deck state, event payload dumps, or other sensitive state. Startup failures are logged structurally before process exit, and legacy `log.Fatal` paths are replaced so bounded telemetry shutdown is not bypassed. Correlation and domain identifiers remain parsed fields, not Loki labels. Alloy keeps only pods in the configured application namespace, including third-party foundation workloads and dedicated Room runtimes; observability and control-plane logs are excluded so Alloy and Loki cannot create a self-ingestion feedback loop. Alloy labels only bounded namespace, service, component, container, and level dimensions; it drops the automatic filename and CRI stream labels, while Loki disables its duplicate automatic `service_name` discovery. Pod names and all command, event, correlation, player, room, game, and tournament identifiers remain structured metadata/query fields. Third-party infrastructure logs in the application namespace may remain raw with the same bounded Kubernetes labels.

Rejected-command audit adapters remain owned by Gateway, Room, and Tournament and keep fail-closed write behavior. They construct `slog.Record` values and call the standardized handler's `Handle` method directly so encoding/stdout failures are returned to the command path; ordinary logger convenience calls cannot provide that guarantee. Required Kubernetes workloads write audits into the same Alloy-collected stdout stream and configure no audit file or volume. Explicit offline/tests may retain file sinks, including their existing command-ID idempotency and conflicting-record checks.

Filesystem-backed Loki storage and Alloy-only log retention are rejected. Grafana is the query and visualization surface; it is not the log store.

## Consequences

The empty-cluster lane must install and ready MinIO before Loki or Tempo, provision both buckets idempotently, then ready the collection backends. It proves that a known structured workload log can be queried through Loki by bounded labels and a parsed correlation field and that a known trace can be queried through Tempo. A MinIO pod-replacement probe must prove both objects remain queryable. Offline checks reject identifier labels and known sensitive-field logging patterns, and validate the 5 GiB claim plus both 24-hour retention configurations. Production deployment requires externally provisioned S3 buckets, scoped credentials, encryption, retention/lifecycle policy, and network access. Local PVC persistence makes no backup, cross-cluster durability, HA, RPO, or RTO claim.

Local images pin exact releases plus verified Linux AMD64/ARM64 OCI index digests: Loki `3.7.3` at `sha256:70b9f699fc9bb868b62f1cfd4f787dfa50242f1fd92e6089787d5d7daea75fe8`, MinIO `RELEASE.2025-04-22T22-12-26Z` at `sha256:a1ea29fa28355559ef137d71fc570e508a214ec84ff8083e39bc5428980b015e`, and MinIO client `RELEASE.2025-04-16T18-13-26Z` at `sha256:aead63c77f9db9107f1696fb08ecb0faeda23729cde94b0f663edf4fe09728e3`. Workloads reference the index so the node runtime selects its architecture. MinIO is a local-only exception because newer source releases no longer have corresponding official server containers; production never deploys this image and uses external S3.
