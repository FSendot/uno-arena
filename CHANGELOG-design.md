# Design Changelog

- **2026-07-15 — The final from-empty kind acceptance passed all 32 live probes.** Starting with no `uno-arena` cluster, the lane recreated the pinned foundation, observability stack, and all eight services; the three business counters each advanced from zero to one, the canonical Gateway/Room/Game Integrity/Ranking trace contained 162 spans, all eight client-parity phases passed, and the corrected Game Integrity two-repository integration passed against real KurrentDB. Tournament store/service integration completed inside the eight-minute package bound, every CDC/adapter/rebuilder/Redis probe passed, gameplay survived a deliberate Alloy outage, and fresh Loki/Tempo/MinIO processes recovered the required log and trace from S3-compatible object storage.

- **2026-07-15 — Game Integrity reconciles ambiguous KurrentDB first appends without replaying writes.** KurrentDB may return an `Aborted` commit-phase timeout when two replicas submit the same deterministic event UUID while the winning first write is still replicating. Game Integrity now treats only first-write deadline, connection, internal-server, aborted, and unavailable outcomes as ambiguous and performs an authenticated read of the committed event. An exact logical event and plaintext proves success; a different candidate proves `conflicting_duplicate`; missing, unreadable, or tampered state preserves the original error and fails closed. The repository never retries the uncertain append. The strict two-repository race integration still requires exactly one winner, one conflict, loser DEK anchoring, exact-retry idempotency, and one committed event.

- **2026-07-15 — The complete kind stack has an explicit single-node memory envelope.** The empty-cluster preflight now requires at least 7 GiB of container-engine memory in addition to 20 GiB of host filesystem headroom. A measured 7.8 GiB node previously entered reclaim/swap pressure during the tournament proof because Kafka silently reserved a 1 GiB minimum heap and the co-located infrastructure could each self-size against large container limits; Kubernetes API directory refreshes then missed deadlines and public Room hops returned 502/503 despite healthy host disk. Kind now uses explicit Kafka, Keycloak, and Debezium heaps, bounded ClickHouse server/caches, conservative 1 GiB native-process ceilings, and 256 MiB Go memory targets for MinIO/Grafana. Heavy single-replica disposable infrastructure uses `Recreate`, preventing a re-apply from temporarily duplicating datastores. The singleton kind Kafka Connect worker also caps scheduled-rebalance recovery at ten seconds instead of retaining the distributed five-minute default after a brief broker stall. Production sizing and overlays are unchanged.

- **2026-07-14 — The single-node kind broker preserves ordering semantics with 44 local partitions instead of 214.** Kind now pre-creates high-volume topics with two partitions, business topics with one, rebuild request/DLQ topics with two, and each Kafka Connect internal topic with one. Keys, immutable partition enforcement, RF1/minISR1, cleanup policy, retention classes, and every recovery proof remain unchanged; the isolated rebuild live probes already use one-partition request topics. AsyncAPI and the production 256/32 planning contract are untouched. This removes 170 idle local partitions and their broker/controller/file overhead from the mandatory observability-plus-tournament acceptance lane.

- **2026-07-14 — The empty-cluster lane fails before creation when the host cannot hold the full local stack.** After deleting only the disposable `uno-arena` cluster, `clean-deploy.sh` now requires 20 GiB of host filesystem headroom before recreating it. A complete ARM64 image set plus workload data consumed roughly 9–10 GiB on the observed Docker VM; starting with only 13 GiB allowed the host to fall below 4 GiB during the tournament proof, causing node-wide filesystem contention, Kubernetes API EOF/timeouts, Room runtime-hop 502s, and a failed recovery read. The check is deliberately after reset so an existing disposable cluster can release its own storage, and before create so insufficient capacity fails clearly instead of producing misleading service failures.

- **2026-07-14 — Kind is the single local deployment and live client-parity lane.** Current runbooks and contracts define one local environment. The clean kind lane owns the deployment-neutral CLI/BFF parity suite through a loopback ClusterIP Gateway port-forward, covering session takeover, Room lifecycle/SSE, stale-sequence rejection, Tournament idempotency, public read schemas, snapshot resynchronization, and terminal spectator denial. Offline fake-BFF and structure checks remain preflight evidence only.
- The public `leaderboard` CLI now satisfies Ranking's required query contract: it defaults to `casual_elo`, accepts the two OpenAPI `boardType` values, and supports bounded `limit` plus opaque `cursor`. Gateway preserves that query string when proxying to Ranking; it no longer turns an otherwise valid client read into an upstream `400`/public `502`.
- Gateway outbound backend HTTP timeout is five seconds in base/production and 12 seconds in kind. The kind value leaves two seconds of response headroom above the Room router's ten-second bound for Gateway forwarding and response handling under single-node load. `GATEWAY_BACKEND_HTTP_TIMEOUT` accepts positive durations through 30 seconds and falls back to five seconds for invalid, zero, negative, or above-maximum input. Kind business bots retain five-second safe-read polling and add a two-second known-result mutation interval while the normal CLI default remains zero. Full observability readiness, the subsequent 15-second exporter settle, mutation pacing, and timeout alignment jointly bound the independent business flows; the settle alone is neither a delivery guarantee nor a substitute for authoritative unknown-outcome recovery.
- Observability persistence now restarts Loki after MinIO is Ready and performs the final recovered-log query through that fresh Loki process, excluding its earlier local TSDB cache/index view from the S3 recovery proof. Offline Istio verification also requires explicit `IfNotPresent` pull policy together with the pinned digest for pilot, CNI, and ztunnel.
- **2026-07-14 — Kind telemetry keeps deterministic command evidence within local node limits.** The trusted kind Gateway always samples mutation roots but applies the production-like `0.001` ratio to read roots, preserving complete causal command traces without tracing every mandatory best-of-three poll. Loki's standard WAL disk backpressure remains enabled at 99 percent for the disposable shared-filesystem kind node while production retains 90 percent, so a local engine image cache above the production threshold cannot reject all application logs while several GiB of bounded lane headroom remain.
- The deliberate Alloy OTLP outage proof now distinguishes during-outage availability from post-restore telemetry recovery: it proves gameplay with zero collector pods, restores Alloy, and waits up to five minutes for the complete observability stack to regain readiness before issuing fresh trace commands. This drains the bounded simultaneous exporter reconnect burst on the shared kind node without coupling application readiness to telemetry or weakening the independent three-minute recovered-trace requirement.
- **2026-07-14 — Empty-cluster observability acceptance protects the authoritative flow from single-node probe load.** The ordered live lane now runs its business-metric and canonical-trace proofs immediately after observability baseline and Debezium connector readiness, before adapter and projection-rebuilder probes accumulate load. Its kind-only bots poll every five seconds by default, reducing repeated authenticated snapshots while retaining the same public Gateway, dedicated Room runtime, CDC, and post-commit metric paths; the cadence remains overrideable. Repository-owned API probes use a kind-only 20-second cadence, the four local Postgres processes and Redis use 15-second readiness plus 30-second liveness checks, and Debezium Connect uses 20- and 30-second checks, preventing kubelet exec-process churn from monopolizing the shared control-plane node while production retains its ten-second application cadence. Between the independent casual and tournament flows, every observability component must recover readiness before the 15-second settle begins; that gate is one part of the bounded flow and is not itself proof that every exporter delivered. Reset-aware Prometheus progress also confirms one additional scrape interval before treating a history-only increase as a process reset, so instant-query lag cannot be mistaken for a duplicate transition. The 188-test Tournament store integration package caps Go test parallelism at two and uses an eight-minute package deadline: a loaded-cluster run reached its final test at 4:58, proving the prior five-minute alarm could interrupt ordinary database work; a repeat passed the store package in 104 seconds and the main package in 11 seconds after settling. The Analytics rebuild proof retries only its read-only ClickHouse queries after a bounded Kubernetes exec transport failure; database creation, Kafka production, and every other mutation remain single-attempt. The persistence probe now captures the current Ready Loki and MinIO pod names and applies bounded retries only to idempotent desired-state Kubernetes mutations, preventing an API timeout that already committed from making a label-based retry delete the replacement. The shared infrastructure readiness gate now treats the successful MinIO bucket Job's TTL garbage collection the same as observability deploy/readiness, while still waiting for it when the object exists.
- The headless client now applies the same bounded safe-GET retry policy to its initial authenticated `whoami` lookup as to Room snapshots and tournament assignments. A transient Gateway `502`, `504`, or timeout during bot setup no longer aborts the next independent acceptance flow; credentials and authoritative failures still fail closed, and no login, registration, or gameplay mutation is retried.
- The local Istio installation budget is ten minutes instead of five, covering exact pinned image pulls on a genuinely cold kind node. Pilot, CNI, and ztunnel renders require their exact digest and explicit kind `IfNotPresent` pull policy. The budget remains overrideable and changes neither image selection nor any production chart behavior.
- The Stage B CLI test harness now gives its local fake Gateway a bounded five-second startup window and one second for the asynchronous SSE board-refresh assertion, eliminating host-load timing failures without changing client retry or production behavior.

- **2026-07-13 — Local business observability acceptance isolates its independent flows.** The kind business-metric probe includes a 15-second exporter settle after full observability readiness and before tournament setup; together with bounded polling, mutation pacing, and timeout alignment this gives the single-node cluster room to drain telemetry without weakening production behavior, but the delay alone makes no delivery guarantee. Its room setup also resnapshots and issues a fresh `JoinRoom` command after an authoritative `stale_sequence` rejection, while retaining the bounded `room_starting` path; neither case replays an uncertain mutation. Gameplay bots reconcile mutation transport `502`, `503`, `504`, or timeout by observing authoritative snapshots for the configured safe-read budget (60 seconds by default, bounded by the overall run deadline) and continue only after a terminal state or sequence advance, including progress from a peer or Room-owned timer. This closes both the terminal-runtime race and a transient router-to-runtime hop loss without replaying unchanged state. The tournament bot reconciles an uncertain `RegisterPlayer` response through the authenticated assignment read model and never replays the mutation. Gateway now distinguishes an authoritative invalid session (`401`) from an Identity dependency or malformed-response failure (`502`), preventing transient validation pressure from falsely invalidating an active client session.

- The live business-metric acceptance gives both casual and tournament best-of-three bot workflows the same 15-minute bound on the resource-constrained kind node, and prints each bot's terminal JSONL records before failing instead of deleting the only diagnostic evidence.
- The centralized log collector now keeps only pods in the configured application namespace, covering repository services, third-party foundation workloads, and dynamic Room runtimes while excluding observability/control-plane logs that could feed Alloy and Loki errors back into Loki. Observability readiness probes use a three-second base timeout and a ten-second kind-only timeout so brief single-node CPU contention does not remove healthy backend endpoints and cascade required-telemetry failures. Grafana disables optional default plugin preinstallation because the version-controlled dashboards need only built-in panels/data sources, removing runtime catalog downloads from deployment. The kind-only Grafana singleton uses `Recreate` and the observability install/upgrade window is ten minutes, avoiding doubled first-start migrations on an already-loaded node; production retains rolling updates and otherwise keeps the base behavior.
- The kind-only Alloy OTLP gateway may burst to two CPUs so an outage replacement can parse its graph and rejoin within the three-minute recovery proof while the single node is processing accumulated telemetry. Its 20m request remains deliberately small, and production retains the chart's independent resource defaults.
- The monolithic kind lane waits 15 seconds after its deliberate Alloy outage/recovery proof before generating the persistence marker, giving application exporters bounded time to discard stale connections without treating the delay as proof of telemetry delivery. The persistence probe registers its marker player exactly once, reports a failed/unknown registration without replay, and then may repeat only an authenticated `whoami` read with the same correlation ID until Loki proves that the recovered log path is live. It also prints its precondition phases and failing line so a future sequencing failure remains diagnosable.
- Debezium Connect and Server remain on `3.6.0.Final`; their repository-wide OCI index pins are refreshed to the registry's current `sha256:8b6267563ceb0cbfe2c3aa5521c4653cbb8bab9d5042e609f2771283f906bada` and `sha256:accbc0d52bcd53f1fe745c2c4957eea8c39be9fd000fb9c20b7d33cbd6c2bfc2`. Both Linux AMD64 and ARM64 children are enforced by the networked portable-image preflight.
- Dedicated Room runtimes in the kind overlay retain six readiness failures but allow ten seconds per health response, preventing a live single endpoint from being withdrawn during observed node-contention spikes that still fit the router's request budget. Business-acceptance bots use a five-second kind-only polling cadence to bound telemetry pressure; production probes and the CLI's default cadence are unchanged.

This changelog records design-package updates made while shaping the architecture checkpoint. It is limited to changes that affect traceability between the original design deliverables and the architecture.

## Empty-Cluster Deployment Acceptance Strategy (2026-07-13)

- ADR-0045 makes the disposable `kind` cluster the single architecture-neutral local deployment and live-test environment. One explicit lane owns the pinned `uno-arena` cluster lifecycle and must deploy infrastructure, services, observability, readiness gates, client parity, functional probes, and business-metric proofs from no cluster. Offline structure and fake-adapter checks remain preflight tools rather than deployment evidence. The local proof makes no production HA, durability, or target-capacity claim.
- The kind-only application profile keeps 50m CPU/64Mi memory requests while allowing repository-owned APIs, workers, and controller-created Room runtimes to burst to 500m CPU/256Mi memory. API and runtime health checks use a three-second timeout and require six consecutive failures before restart, preventing single-node acceptance traffic and telemetry load from becoming false liveness failures. The kind Alloy Logs collector can likewise burst to 500m CPU while initializing readable node logs; base, staging, and production defaults are unchanged.
- Local Debezium Kafka Connect uses its bundled plugins' ServiceLoader metadata instead of the legacy hybrid reflective second scan, keeping restart discovery within the bounded probe window during single-node contention without changing connector configuration or CDC semantics.
- The clean deployment order is foundation/Istio, local-only MinIO plus idempotent Loki/Tempo bucket provisioning, the repository-owned portable `uno-arena-observability` Helm release, and then application Helm releases, with each stage readied before dependents. Kind values target MinIO; production values only describe external S3 and Secret references and do not provision cloud resources.
- ADR-0046 standardizes centralized logs on Alloy → monolithic Loki TSDB → Grafana. Loki uses the same S3-compatible storage contract in every environment: local `kind` supplies MinIO and a dedicated bucket, while production defaults to an external S3 bucket with environment-owned endpoint, bucket, region, TLS, credentials, encryption, and lifecycle policy. Filesystem-only Loki and Alloy-only retention are rejected.
- Local MinIO stores Loki objects on a Kubernetes PVC so logs survive MinIO pod replacement while the cluster exists. Monolithic Loki enables its WAL, explicitly ships the active TSDB index to S3-compatible storage, and flushes on graceful shutdown; persistence acceptance creates a fresh player through the public CLI with a dedicated low-volume correlation marker, invokes the supported ingester shutdown with full-process termination, and waits for Loki to cut its active 15-minute TSDB head plus synchronously upload the index before deleting the exact restarted Pod. A distinct empty-work-volume replacement must query the correlated application log from S3 before MinIO replacement; after MinIO returns Ready, Loki is replaced again and the final query must pass through that fresh cache/index path. Tempo independently preserves the canonical cross-context gameplay trace. Deleting the disposable kind cluster intentionally deletes the volume and logs; local observability makes no backup, cross-cluster durability, HA, RPO, or RTO claim.
- The local observability budget is one 5 GiB MinIO PVC shared by separate `loki-logs` and `tempo-traces` buckets. Loki's Compactor and Tempo independently enforce 24-hour retention without a competing MinIO lifecycle rule; production capacity and retention remain environment-owned.
- Repository-owned application and business metrics use the OpenTelemetry Go Meter API. Each process exposes a custom-registry OpenTelemetry Prometheus exporter on named internal port `metrics` at `:9090/metrics`, with explicit underscore-and-type-suffix translation; standalone Prometheus scrapes it directly, while Alloy remains outside the pull-based metrics path. Stable APIs use EndpointSlice discovery through their existing Services; service-less workers and dedicated Room runtimes use direct pod discovery, including unready targets. Mutually exclusive scrape modes prevent duplicates, and no metrics-only worker Services or Prometheus Operator resources are introduced. Local scrape/evaluation cadence is 15 seconds with a 5-second timeout and a 45-second acceptance polling budget; production may override cadence. Exposition contract tests pin exact names, types, help text, and bounded labels across exporter upgrades. Mimir, Alertmanager, and the full `kube-prometheus-stack` are outside the local acceptance scope.
- The required business dashboard proves three context-owned post-commit OpenTelemetry counters. Instruments `unoarena.players.created`, `unoarena.games.completed`, and `unoarena.tournaments.completed` export exactly as `unoarena_players_created_total`, `unoarena_games_completed_total`, and `unoarena_tournaments_completed_total`. Each owner records an unlabeled zero after telemetry startup so an empty cluster exposes the descriptor and a real pre-flow baseline without claiming a transition; only post-commit hooks add one. Rejections, failed commits, and idempotent replays are excluded. Each uses unit `1`, exact context-specific help text, and no instrument labels; bounded service, namespace, component, pod, and instance labels come only from target discovery. These process-local counters are operational evidence rather than authoritative accounting; durable context state/events remain truth, dashboards use `sum(increase(...[$__range]))` across replicas and resets, and acceptance proves a positive increase between pre-flow and post-flow scrapes. A separate Kubernetes dashboard carries workload readiness, unavailable replicas, restarts, and pod/container CPU and memory.
- The cluster dashboard sources object state from kube-state-metrics and pod/container CPU, memory, network, and filesystem series from authenticated kubelet/cAdvisor scrapes. Metrics Server is unnecessary, while local node-exporter is excluded because kind's Docker-contained node would make host-level panels misleading.
- Local Prometheus uses a 5 GiB PVC, 24-hour retention, and a storage ceiling below filesystem capacity. Its history survives pod replacement but is intentionally removed with the kind cluster; production metrics retention remains environment-owned.
- Grafana is stateless and ClusterIP-only: version-controlled ConfigMaps provision non-editable Prometheus/Loki data sources and business/cluster dashboards, local credentials come from the Kubernetes Secret, access uses a loopback-only port-forward, and live acceptance verifies provisioning through the HTTP API without manual UI setup.
- Alloy, Loki, Tempo, MinIO, Prometheus, kube-state-metrics, and Grafana run in a dedicated `observability` namespace with separate service accounts. Collection components receive only scoped read permissions, storage/UI components disable token automount, application Secrets are inaccessible, and `/metrics` remains an internal Prometheus-only scrape surface. Istio authorization and Kubernetes ingress policies enumerate the exact client identity and port for Loki, Tempo, Prometheus, kube-state-metrics, MinIO, OTLP ingestion, and application metrics instead of granting observability-namespace-wide access.
- Alloy log and trace roles use separate identities: `alloy-logs` has only pod-metadata reads plus a read-only `/var/log/pods` host mount. Because that container-runtime directory is `0750 root:root`, this non-root UID/GID 473 node collector alone receives supplemental GID 0 to traverse and read it; `runAsNonRoot`, `RuntimeDefault` seccomp, a read-only root filesystem, disabled privilege escalation, and all capabilities dropped remain enforced. `alloy-otlp` remains non-root, disables token automount, and has no Kubernetes API access. Its private `4317`/`4318` Service is protected by strict Ambient mTLS, default-deny authorization, and matching NetworkPolicy for dedicated application service accounts without shared telemetry credentials; Loki and Tempo reject direct application ingestion.
- Kind installs Istio Ambient and enrolls both `uno-arena` and `observability` under strict east-west mTLS and default-deny L4 authorization. Only the Prometheus service-account principal may scrape application port `9090`, and acceptance proves both that allow and an unauthorized-workload denial. Kind uses those SPIFFE-scoped mesh controls without adding a second CNI; production chart renders enable non-overlapping per-workload NetworkPolicies, including HBONE `15008` and link-local kubelet health-probe exceptions. The local plaintext exception is narrowed to documented application-layer database TLS and disposable credentials; it never permits plaintext mesh transport.
- Repository-owned Go services/workers standardize on one-line `log/slog` JSON. Loki labels are limited to bounded namespace/service/component/container/level dimensions: Alloy drops automatic filename and CRI stream labels, and Loki disables duplicate `service_name` discovery. All identifiers remain parsed fields. Credentials, tokens, private gameplay state, encryption material, connection strings, and payload dumps are forbidden, while third-party logs may remain raw.
- The application log schema now pins UTC RFC3339Nano timestamp, lowercase level, service/component/environment/version/instance identity, bounded snake-case event, optional message, and active trace/span IDs. Domain identifiers keep canonical camelCase spelling, sanitized failures use only `error`, reserved fields cannot be overridden, and source/stack dumps plus shutdown-bypassing `log.Fatal` paths are excluded.
- Rejection audits remain context-owned and fail closed by invoking the standardized `slog` handler directly, allowing JSON/stdout errors to propagate while preserving command-ID idempotency/conflict rules. Durable rejected outcomes are committed only after the audit succeeds; retries therefore repair audit failures, while the audit seam remains intentionally at-least-once if the context commit fails after emission. Stable duplicate comparison excludes the emission timestamp. Required Kubernetes workloads emit audits into Alloy's stdout path and reject audit-file configuration or volumes; file sinks remain offline/test-only.
- ADR-0047 makes distributed tracing part of deployment acceptance: Go services/workers use OpenTelemetry and W3C Trace Context, a dedicated Alloy gateway receives OTLP and forwards to monolithic Tempo, local traces use a dedicated MinIO bucket while production defaults to S3, and Grafana provisions Tempo plus trace-to-log links. A separate OpenTelemetry Collector is intentionally unnecessary.
- Transport/datastore tracing uses established adapter instrumentation: `otelhttp`, franz-go `kotel`, `redisotel`, and `otelpgx`, with `otelhttp` also wrapping Analytics' ClickHouse transport. Game Integrity owns meaningful manual KurrentDB spans, while bounded contexts define command/commit/publication/worker spans and exclude SQL text, Redis keys, payloads, credentials, and identifier metric labels. Every `otelpgx` adapter uses a bounded context-owned query span name and disables SQL statement attributes, and every `redisotel` adapter disables raw command statements; live trace acceptance rejects SQL-bearing query names and datastore statement attributes.
- OpenTelemetry resource identity is uniform across signals: canonical stable service name, `service.namespace=uno-arena`, build revision, deployment environment, pod-UID instance ID, and a bounded API/worker component. Roles share a service name; domain/correlation identifiers remain controlled span/log fields and never become resource attributes or metric labels, while Prometheus keeps Kubernetes target metadata separate.
- OpenTelemetry mechanics are isolated in an infrastructure-only `platform/telemetry` Go module that owns resource, provider, propagation, exporter, bounded-queue, shutdown, and `log/slog` JSON-handler setup. The handler applies shared identity and active trace/span correlation while stdout remains the Alloy log path; no OTLP log exporter is introduced. Bounded contexts still own their instruments, span/attribute vocabulary, post-commit recording seams, log events/fields/sanitization, and log placement; no domain types or business metric catalog move into the platform module, and `shared` remains limited to standard-library helpers.
- The telemetry module exposes one deep interface: start once before adapter construction to receive the logger, explicit providers, and propagator, then stop business work before one bounded shutdown flush. Required startup validates configuration and synchronously binds metrics; disabled mode returns no-op providers with the same JSON logger. Exporter/registry/queue internals stay private. The only global provider is the same MeterProvider registered once through OpenTelemetry's standard seam for SDK self-observability; tracer/propagator remain explicit, and the standard global error handler emits sanitized structured failures.
- Trace sampling is environment-owned: kind uses `x_parentbased_mutations` at the trusted Gateway so mutation roots are always sampled for complete command evidence while read roots use the production-like `0.001` ratio; other long-running services and production default to configurable `parentbased_traceidratio=0.001` for autonomous roots. Child spans follow the Gateway decision, short-lived rebuild acceptance Pods may sample always-on, and the local Alloy gateway batches without tail sampling. This keeps mandatory best-of-three polling within the single-node acceptance budget without relying on untrusted client sampling flags.
- The single-node kind overlay defers Tempo compaction measurement and scheduling for one hour, requires a 1,000-block compaction group so the scheduler's startup measurement cannot saturate the acceptance node, cuts live-store blocks every ten minutes, and refreshes the block list every minute so completed S3 blocks become queryable inside bounded persistence acceptance. Its Alloy-to-Tempo exporter uses a bounded 2,048-request queue with two consumers, a 30-second request timeout, and ten-minute capped retries so short node-contention windows do not cascade into application readiness failures; Tempo may use two CPUs locally. The base and production chart retain normal 2–4 block groups, 30-second live-store blocks, five-minute block polling, one-minute compaction measurement, a 30-second minimum cycle, and a one-hour work-prune age.
- Monolithic Tempo owns a separate 1 GiB WAL/work PVC. Acceptance waits until Tempo's block-only query path proves the trace is in S3, then recreates the WAL PVC and replaces Tempo and MinIO; the trace must remain queryable afterward. Both PVCs remain disposable with the kind cluster and make no local HA, backup, RPO, or RTO claim.
- All generic third-party workloads pin an exact release plus a registry-verified multi-architecture OCI index containing Linux AMD64 and ARM64 children; manifests reference the index and let the node runtime resolve its child instead of embedding or pre-staging ARM64. This covers Istio 1.30.2, the observability stack, foundation stores, and Debezium. KurrentDB alone uses a reviewed AMD64/ARM64 digest map because no suitable shared index exists. Repository-built kind images target the detected node architecture. MinIO remains local-only; production uses external S3.
- CDC tracing uses nullable `traceparent`/`tracestate` columns committed in every Kafka-bound and Room realtime outbox. Debezium Connect maps them to Kafka headers, Debezium Server maps them to Redis metadata, and direct Kafka clients use the same W3C headers. Baggage is never persisted, and the undeployed initial schemas are corrected in place rather than adding upgrade migrations.
- The canonical live trace proof is the accepted casual-game command that emits `GameCompleted`: one causal trace crosses Gateway and Room HTTP, Room command/commit, Game Integrity append, Room outbox, Debezium-mediated Kafka context transport, and Ranking consume/commit. Debezium need not emit a span; Tempo returns the trace and Loki returns logs by parsed trace/correlation identifiers without identifier labels.
- OTLP export is asynchronous and bounded: queues, retries, timeouts, and shutdown flushes cannot block business work or application readiness. Alloy/Tempo outage may drop spans but must emit self-observability signals; a held batch may still fail through its stale gRPC connection after Alloy becomes Ready, so acceptance removes/restores Alloy, proves gameplay continuity, and issues bounded fresh commands until one complete new trace proves exporter reconnection.
- Headless gameplay bots reconcile transport-unknown command outcomes (`502`, `503`, `504`, or timeout) by reading fresh authoritative Room state for the configured safe-read budget (60 seconds by default, bounded by the overall bot deadline) and continuing only after a terminal state or sequence advance. This covers delayed dedicated-runtime replacement between best-of-three games. Snapshot and assignment GETs boundedly retry their safe transient transport failures as well as `503 room_starting`; failed mutation envelopes are never replayed against unchanged state, and failed reconciliation or failure to reach a terminal Room/Tournament remains non-zero.
- The Room router keeps a newly Kubelet-Ready dedicated runtime out of its bounded pod directory for a three-second data-plane warmup, closing the first-request race between container readiness and Ambient enrollment while preserving `503 room_starting` until routing is admitted.
- Required workloads enable the pinned SDK's official experimental self-observability and nonblocking batch processor with queue 2048, batch 512, five-second schedule/export deadlines, and retries bounded by that deadline. Prometheus carries queue size/capacity/processed/`queue_full` signals, while a local wrapper counts and logs sanitized backend export failures. Experimental metric names remain internal and pinned/tested rather than business contracts.
- Telemetry startup distinguishes explicit offline `disabled` from Kubernetes `required` mode. Every Helm API/worker must require telemetry and static acceptance rejects disabled or unspecified workloads; invalid resource/exporter configuration or metrics-port binding fails startup. After successful initialization, backend outages remain bounded and never gate application readiness.
- The telemetry configuration contract uses `TELEMETRY_MODE`, existing service/environment identity, immutable service version, explicit bounded component, Downward-API pod UID, standard OTLP endpoint/protocol and trace-sampler variables, and `METRICS_ADDR=:9090`. Helm supplies the complete contract to APIs, workers, and generated Room runtimes; component identity is never inferred from pod names.
- Telemetry source modules are pinned at their current compatible releases while the repository remains `go 1.26.0`/`go1.26.5`: OpenTelemetry/SDK/OTLP `v1.44.0`, Prometheus exporter `v0.66.0`, `otelhttp v0.69.0`, `kotel v1.7.0`, `redisotel v9.21.0`, `otelpgx v0.11.1`, and Prometheus client `v1.23.2`. All `go-redis` consumers align on `v9.21.0`; source-module minimum Go declarations do not constrain independently pinned ARM64/AMD64 OCI images.

## Runtime Routing and Client Parity Contract (2026-07-13)

- Stale room-sequence command outcomes use HTTP `409 Conflict` while preserving the structured rejected command result. Accepted commands and non-stale domain rejections remain HTTP `200`.
- ADR-0042 defines event-driven tournament assignment readiness: Room atomically emits `RoomRuntimeReady` on first runtime readiness, Tournament withholds `roomId` until consuming it, and later runtime replacement uses bounded client `room_starting` retry without revoking the assignment.
- ADR-0043 defines authorization-first Room routing: the stable boundary validates the original route-specific caller and BFF-derived player principal before pod lookup, then upgrades only authorized traffic to the scoped router credential and pinned generation.
- The CLI uses a configurable 60-second default budget for safe `room_starting` reads, honoring `Retry-After` with jitter and a five-second interval cap while never retrying mutations after unknown outcomes.
- ADR-0044 separates Room maintenance roles: dedicated runtimes and routers perform no global scans; the timer worker rebuilds under one context lease whose Postgres-owned completion generation fences contemporaneous replicas without cross-pod clock comparisons, and bounded integrity-reconciler replicas claim repair work with `FOR UPDATE SKIP LOCKED`.

## Client Parity and Dedicated Room Runtime Topology (2026-07-13)

- Client playability and bot selection now apply the Room contract during a pending penalty: only an ordinarily legal `Draw Two`, or a `Wild Draw Four` when the hand contains no non-wild active-color card, is selectable. The server remains authoritative.
